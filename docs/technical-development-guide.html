Project Guide

How the Vancouver housing prediction project works

This document explains the project from both perspectives: how a software engineer should understand the runtime, API, and code organization, and how a data scientist should understand the dataset, model choices, validation, and current limitations.

React + Vite + Tailwind Express 5 + Zod Python Flask model service XGBoost / Random Forest Vancouver only
Current Project

What this demonstrates

The project estimates Vancouver home listing prices with property-type-specific ML models, then shows how the prediction can be wrapped in a small user-facing app. The renovation section is intentionally rule-based because the available dataset does not support causal uplift modeling.

SWE DS Prototype
Truthfulness Boundary

What it should not claim yet

The base model currently predicts listing price, not verified sale price. The live uplift layer is rule-based, not a trained causal renovation model. A true ML uplift model still requires licensed, property-level before/after resale data.

Treat outputs as directional planning estimates, not guarantees or investment advice.
Architecture

Three-tier runtime shape

The project is intentionally split into a web client, a public API layer, and a Python model service. This keeps frontend UX, API validation/orchestration, and ML code independent enough to evolve separately.

Runtime Flow

How requests move through the system

Flow What happens Main files
Estimate The user enters postal code, property type, square footage, beds, baths, optional property tax, and optional known value. React calls POST /api/estimate. Express validates the shape and proxies to Flask /estimate. Python loads the base model bundle and returns current list-value estimate plus trust metrics. App.tsx, EstimatePage.tsx, generated.ts, index.ts, model.ts, service.py
Simulate The user toggles improvements. React calls POST /api/simulate. Express validates, refreshes the base estimate, and runs the rule-based uplift engine to return uplift, confidence range, final guardrailed value, and rule explanations. SimulatePage.tsx, ImprovementFlagPicker.tsx, schemas.ts, model.ts, ruleBasedUplift.ts
Plan The user sets target price, budget, and timeline. Express scores candidate improvements through the same rule-based uplift engine, chooses actions that fit budget/time, and returns phases plus target realism status. PlanPage.tsx, model.ts, data.ts, ruleBasedUplift.ts

SWE View

Frontend architecture

The frontend is a small React website with three product routes. It keeps state local and simple: one persisted property profile, one persisted improvement-flag list, and live estimate state.

State model

  • property: postal code, type, sqft, beds, baths, tax, known value.
  • plannedFlags: selected improvement toggles.
  • estimate: current base model response.
  • requestError: API/model-service error surfaced to the user.

Routes

  • /: product landing page.
  • /estimate: current listing-value estimate.
  • /simulate: rule-based uplift simulation.
  • /plan: target, budget, timeline, and recommendations.
SWE tip: if request shapes change, update schemas.ts, types.ts, client.ts, and home-value-planner.openapi.yaml in the same commit.

API server responsibilities

The Express server is intentionally thin but important. It protects the Python service from malformed input, hides model-service URLs from the browser, and centralizes sale-plan orchestration.

Endpoint Input Output
GET /health none API status and mode.
POST /api/estimate PropertyInput Base estimate, confidence range, model quality, drivers, market context.
POST /api/simulate PropertyInput + plannedFlags + horizonMonths Uplift, final value, evidence level, confidence range, top uplift drivers.
POST /api/plan SimulateRequest + targetPrice + budget + timelineMonths Target status, achievable value, chosen recommendations, phases.
DS + SWE View

Python model service

The Python service handles model loading, training, feature engineering, inference, and health payloads. It is the live runtime for the base estimate. The older Python uplift path remains as research scaffolding, while simulate and plan now use the Express rule-based uplift engine.

Base model

Trains on the Vancouver subset of data_bc.csv. It filters to supported property types, engineers postal/FSA and location features, trains on log(price), compares XGBoost and Random Forest, and selects the best model per property type by CV MAE.

Rule-based uplift

Uses deterministic cost-recovery, feasibility, property-type, timeline, and local ceiling rules. It is explainable and useful for scenario planning, but it is not trained on observed renovation resale outcomes yet.

DS View

Data science understanding

Layer 1: base listing-value model

Concept Current implementation
Target Listing price from Vancouver rows in data_bc.csv.
Scope Vancouver only; property types are Condo, Detached, Townhouse, and Duplex.
Features Living area, bedrooms, bathrooms, property tax, latitude, longitude, polynomial lat/lon features, postal FSA, and KMeans submarket cluster.
Models XGBoost and Random Forest candidates, trained per property type.
Validation Outlier removal, train/holdout split, adaptive cross-validation, bootstrap intervals.
Output Base value, confidence range, price per sqft, market context, drivers, and model-quality payload.

Layer 2: rule-based uplift

Concept Current implementation
Target Estimated value added from selected improvements using published cost-recovery assumptions, Vancouver feasibility rules, and market headroom.
Improvement flags Kitchen, bathrooms, legal suite, energy upgrades, curb appeal, permit issues, deferred maintenance, roof/systems.
Evidence levels rule-based in the live product. observed, hybrid, and proxy-heavy are reserved for future/archived uplift experiments.
Guardrail Final value is guardrailed against a practical market ceiling.
Known weakness Rule-based uplift is not equivalent to a true causal renovation value model.
DS priority: the biggest next leap is acquiring actual transaction sale data and rebuilding both Layer 1 and Layer 2 around sale outcomes instead of listing/proxy outcomes.
Contracts

How to keep interfaces aligned

Contract drift was one of the biggest project risks. The current runtime source of truth is the Zod schema in artifacts/api-server/src/schemas.ts. The TypeScript frontend types and OpenAPI snapshot should mirror it. 

Update order for API changes:
1. artifacts/api-server/src/schemas.ts
2. artifacts/api-server/src/model.ts response types
3. artifacts/home-value-planner/src/types.ts
4. artifacts/home-value-planner/src/api/client.ts
5. artifacts/openapi/home-value-planner.openapi.yaml
6. affected React pages

Known Limitations

What still needs caution

Modeling limitations

  • Base estimate predicts listing price, not actual sale price.
  • Uplift is rule-based and not trained on observed resale outcomes yet.
  • Renovation rules cannot guarantee completion quality, buyer taste, or permit outcomes.
  • Property tax is highly missing in the listing dataset.

Engineering limitations

  • model.ts mixes transport and planner orchestration.
  • The archived uplift_service.py research module is too large and should be split if revived.
  • The DB layer is provisioned but unused.
  • Model artifacts are local pickle files, not a robust registry.
Roadmap

Recommended next improvements

Priority Improvement Why it matters
1 Acquire and integrate BC Assessment actual sales data. Turns the base model from listing-price prediction into sale-price prediction.
2 Build true before/after resale uplift labels. Moves uplift from transparent rules to observed or hybrid evidence.
3 Split uplift_service.py into ingestion, features, training, inference, and evidence modules. Reduces risk and makes DS work easier to test.
4 Split model.ts into model-service client and planner orchestration modules. Makes the API easier to maintain and test.
5 Add unit tests for schemas, planner scoring, flag mapping, and error handling. Protects the app from regressions as the model evolves.
6 Add persistent saved scenarios with the dormant Drizzle/Postgres layer. Turns the demo into a more complete product workflow.
Bottom Line

How to describe the project today

This is a Vancouver-only data science project with a real React/Express/Python demo layer, a validated base listing-price model, and a transparent non-ML uplift planner. The strongest resume story is the base predictive modeling workflow plus the decision not to overclaim uplift modeling without the right labels.
